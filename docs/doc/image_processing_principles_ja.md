## 1. パイプライン全体像

画像処理パイプラインは、MIPI CSI-2 受信器（OV5640 カメラデータを24ビットRGBピクセルストリームに変換）と AXI4-Stream ビデオブリッジ（DDR キャプチャおよび HDMI 表示用の VDMA に接続）の間に挿入される。全処理は `sysclk`（125 MHz）ドメインで動作する。

```
video_pixel [23:0] (rgb565_gray_unpack から)
       |
       v
  [PRE] axis_rgb_prefilter ---- 3x3 デノイズ (メディアン / ガウシアン) + 点処理
       |
       +------ A 枝 -----------+------ B 枝 (S1) ----------+
       |                        |                            |
       v                        v                            |
  axis_rgb_conv3x3         axis_rgb_conv5x5                  |
  (任意 3x3 カーネル)      (任意 5x5 カーネル)               |
       |                        |                            |
       +----> dog_combine <-----+                            |
       |      (alpha*A - beta*B + offset)                    |
       |                        +---> axis_rgb_conv5x5_sep (S2) --- 分離型 5x5
       |                        |          |
       |                        |     axis_rgb_conv5x5_sep (S3) --- 分離型 5x5
       |                        |          |
       v                        v          v
  +----- 最終マルチプレクサ (cfg_proc_op) ---+
  |  op 0-11:  conv3x3 出力 (単一カーネル or 点処理)        |
  |  op 12:    DoG コンバイナ出力 (二重カーネル)             |
  |  op 13:    conv5x5 出力 (カスケード段1, 実効 5x5)       |
  |  op 14:    S2 出力 (カスケード段2, 実効 9x9)            |
  |  op 15:    S3 出力 (カスケード段3, 実効 13x13)          |
  +----------------------------------------------------------+
       |
       v
  [POST] axis_rgb_proc_slot ---- 点処理 (反転, グレースケール, 二値化 等)
       |
       v
  [DITHER] axis_rgb_dither ---- 順序(Bayer) / ランダム(LFSR) ビット深度ディザ
       |
       v
  axis_video_bridge --> VDMA --> HDMI
```

### スロット契約

全処理モジュールは同一のインタフェースを共有する:

```
入力:  {pixel[23:0], valid, sof, eol, eof, err}
出力:  {pixel[23:0], valid, sof, eol, eof, err}
```

- `pixel[23:0]` = `{R[7:0], G[7:0], B[7:0]}` (RGB888)
- `valid` = このクロックサイクルでピクセルが有効
- `sof` / `eof` = フレーム開始 / 終了
- `eol` = ライン終了 (列639)
- `err` = 上流エラー

全モジュールに `ENABLE` パラメータがあり、0 のときは純粋なワイヤスルー（ロジックコスト 0）。有効時は全モードで**固定レイテンシが同一**なため、ランタイムでモードを切り替えてもマーカーのずれが発生しない。

## 2. 点処理 (PRE / POST)

### 2.1 原理

**点処理 (point operation)** は各ピクセルをそのピクセルの値のみで独立に変換する — 空間近傍は不要。最もシンプルかつ低コストな画像処理（純粋な組合せ論理、ラインバッファ不要、DSP不要）。

### 2.2 処理一覧

| op | 名称 | 式 | 用途 |
|----|------|------|------|
| 0 | パススルー | `out = in` | 無加工 |
| 1 | 反転 | `out = {255-R, 255-G, 255-B}` | ネガ画像 |
| 2 | グレースケール | `Y = G; out = {Y, Y, Y}` | 輝度近似（緑チャネル ≈ 輝度の59%） |
| 3 | BGR スワップ | `out = {B, G, R}` | 赤/青チャネル交換 |
| 4 | 二値化 | `out = (G > T) ? 白 : 黒` | 閾値 T による二値セグメンテーション |
| 5/6/7 | R/G/B のみ | 他の2チャネルをゼロに | カラーチャネル分離 |

グレースケール近似はGチャネルをそのまま使う。重み付き和 `0.299R + 0.587G + 0.114B` を使わないのは、乗算器が Z-7020 の配線混雑を悪化させるため。

点処理は PRE（畳み込み前）と POST（畳み込み後）の2箇所に配置されており、「エッジ検出→二値化」のような連鎖が可能。

## 3. 空間畳み込み — 原理

### 3.1 畳み込み演算とは

2次元離散畳み込みは、小さな**カーネル**（重み行列）を画像のスライディングウィンドウに適用する:

```
out(x, y) = Σ(i,j) kernel(i, j) * image(x+i, y+j)
```

3x3 カーネルの場合、1ピクセル・1カラーチャネルあたり 9 回の積和演算が必要。結果は正規化（右シフト）され [0, 255] にクランプされる。

カーネルの係数を変えることで異なる効果が得られる:

| カーネル | 重み | 効果 |
|----------|------|------|
| 恒等 | `[0,0,0; 0,1,0; 0,0,0]` | 変化なし |
| ガウシアンぼかし | `[1,2,1; 2,4,2; 1,2,1] / 16` | ローパス平滑化 |
| シャープ化 | `[0,-1,0; -1,5,-1; 0,-1,0]` | エッジ強調 |
| Sobel-X | `[-1,0,1; -2,0,2; -1,0,1]` | 水平勾配 |
| エンボス | `[-2,-1,0; -1,1,1; 0,1,2]` | レリーフ効果 |

### 3.2 ソフトウェア vs ハードウェア — FPGA畳み込みが根本的に異なる理由

ソフトウェア（CPU/GPU）では、3x3 畳み込みは単なるネストしたループで書ける:

```python
for y in range(H):
    for x in range(W):
        acc = 0
        for i in range(-1, 2):
            for j in range(-1, 2):
                acc += kernel[i+1][j+1] * image[y+i][x+j]
        out[y][x] = acc >> shift
```

CPU はメモリ上の画像全体にランダムアクセスできるため、任意のピクセルをいつでも取得できる。このループは1ピクセルずつ逐次処理する。

FPGA のストリーミングパイプラインでは、状況が根本的に異なる:

1. **ランダムアクセス不可**: ピクセルは1クロックサイクルに1つ、ラスタスキャン順（左→右、上→下）で到着する。任意にインデックスできる「フレームバッファ」は存在しない。
2. **スループット要件**: 640x480 @ 30fps では約 9.2 Mpixel/s でピクセルが到着する。パイプラインは入力ピクセルごとに1つの出力ピクセルを毎クロック生成し続けなければならない。
3. **並列性**: CPU のループが 9 回の乗算を逐次実行するのに対し、FPGA は 9 回の乗算を専用ハードウェア（DSP48 スライス）で**同時に**実行する。これは空間的並列性 — 「ループ本体」が 9 個の物理乗算回路に展開される。

根本的な課題: **1クロックに1ピクセルしか受信しないのに、3x3 の近傍ピクセルにどうアクセスするか？**

### 3.3 ラインバッファ — ハードウェアでスライディングウィンドウを形成する

#### 3.3.1 根本的な問題 — 「過去のピクセル」をどう覚えるか

3x3 畳み込みを計算するには、注目ピクセルの**上下左右斜め**の合計9ピクセルが同時に必要。しかしカメラからピクセルが届く順番はこうなっている:

```
画像:                     到着順（1クロックに1つ）:
  行0:  A  B  C  D         t=0: A → t=1: B → t=2: C → t=3: D →
  行1:  E  F  G  H         t=4: E → t=5: F → t=6: G → t=7: H →
  行2:  I  J  K  L         t=8: I → t=9: J → t=10: K → t=11: L
```

例えば t=6 で G が到着した瞬間に、G を中心とした 3x3 ウィンドウを作りたい:

```
必要なウィンドウ:
  B  C  D     ← 行0（B は5クロック前、C は4クロック前、D は3クロック前に通過済）
  F  G  H     ← 行1（F は1クロック前、G は今、H はまだ来ていない）
  J  K  L     ← 行2（まだ全部来ていない）
```

問題が2つある:
1. **B, C, D はとっくに通り過ぎた** — 保存していなければ取り出せない
2. **H, J, K, L はまだ来ていない** — 未来のピクセルは待つしかない

問題2の解決は単純: **出力を遅延させる**。今届いたピクセルではなく、1ライン+1列前のピクセルを中心とした畳み込みを出力する。つまり t=10 で K が届いた瞬間に、ようやく G を中心としたウィンドウが完成して出力できる。

問題1が本題: **過去に通過したピクセルをどう保存するか？** → これが**ラインバッファ**の役割。

#### 3.3.2 核心のアイデア — 「1行分のメモリ」で行をまたぐ

ラインバッファの考え方は極めてシンプル:

> **「各列のピクセルは、次の行の同じ列のピクセルが届くまでの間、保存しておく」**

幅4ピクセルの画像で説明する。lbA は「1行前のピクセルを覚えておく棚」、lbB は「2行前のピクセルを覚えておく棚」:

```
lbA:  [棚0] [棚1] [棚2] [棚3]     ← 4列分の棚（＝画像幅分）
lbB:  [棚0] [棚1] [棚2] [棚3]     ← 同じく4列分
```

##### t=0: ピクセル A（行0, 列0）が到着

まだ何も保存されていない（初期状態は不定）。A を lbA[0] に入れる:

```
lbA:  [ A ] [   ] [   ] [   ]
lbB:  [   ] [   ] [   ] [   ]
出力: なし（まだ行が足りない）
```

##### t=1: ピクセル B（行0, 列1）が到着

B を lbA[1] に入れる:

```
lbA:  [ A ] [ B ] [   ] [   ]
lbB:  [   ] [   ] [   ] [   ]
出力: なし
```

##### t=2, t=3: C, D が到着 → lbA に格納

```
lbA:  [ A ] [ B ] [ C ] [ D ]     ← 行0 が全部入った
lbB:  [   ] [   ] [   ] [   ]
出力: なし（まだ1行しかない）
```

##### t=4: ピクセル E（行1, 列0）が到着 ★重要★

列0に再び到着。lbA[0] には前の行の A が入っている。ここで**玉突き**が起きる:

```
手順:  1. prev1 = lbA[0] を読む → A を取得（1行前の列0）
       2. lbB[0] ← lbA[0] の旧値（= A）を移動
       3. lbA[0] ← E（今のピクセル）を格納
```

```
lbA:  [ E ] [ B ] [ C ] [ D ]     ← 列0 が E で上書きされた
lbB:  [ A ] [   ] [   ] [   ]     ← A が lbB に押し出された
```

いま同時に3つの値が使える: in_pixel = E、prev1 = A。つまり**列0における行0と行1のピクセルが同時に取れた**。

##### t=5: ピクセル F（行1, 列1）が到着

同じ玉突き:

```
lbA:  [ E ] [ F ] [ C ] [ D ]
lbB:  [ A ] [ B ] [   ] [   ]
同時に取れる値: in_pixel=F, prev1=B（1行前の列1）
```

##### t=6, t=7: G, H が到着

```
t=6: lbA:  [ E ] [ F ] [ G ] [ D ]   lbB:  [ A ] [ B ] [ C ] [   ]
     in=G, prev1=C
t=7: lbA:  [ E ] [ F ] [ G ] [ H ]   lbB:  [ A ] [ B ] [ C ] [ D ]
     in=H, prev1=D
```

##### t=8: ピクセル I（行2, 列0）が到着 ★3行分が揃う★

列0に3度目の到着。今度は lbA にも lbB にも値がある:

```
手順:  1. prev1 = lbA[0] を読む → E（1行前=行1の列0）
       2. prev2 = lbB[0] を読む → A（2行前=行0の列0）
       3. lbB[0] ← lbA[0] の旧値（= E）
       4. lbA[0] ← I（今のピクセル）
```

```
lbA:  [ I ] [ F ] [ G ] [ H ]
lbB:  [ E ] [ B ] [ C ] [ D ]
同時に取れる値: in_pixel=I, prev1=E, prev2=A  ← 列0における行0,1,2 の3行分！
```

#### 3.3.3 ラインバッファの本質 — なぜ「1行分」なのか

ここまでの例で分かるように:

- lbA は「1行前」のピクセルを列ごとに覚えている
- lbB は「2行前」のピクセルを列ごとに覚えている
- 毎回同じ列のアドレスにアクセスするので、次にその列の新しいピクセルが来たときに古い値を読み出せる

なぜ lbA の大きさが「画像幅（640エントリ）」なのか？ → **ピクセルは左から右に1つずつ来るので、同じ列に再び来るのは1行分（640ピクセル）後**。640個の棚があれば、全列の「1行前の値」を保持できる。

図解:

```
画像幅 = 4 の場合:

到着順:  行0: [A B C D] → 行1: [E F G H] → 行2: [I J K L]

lbA の棚の中身の変化:
  t=0〜3 (行0処理中):  lbA = [A, B, C, D]  ← 行0が溜まる
  t=4〜7 (行1処理中):  lbA = [E, F, G, H]  ← 行1に上書きされていく
                                               (行0は lbB に押し出される)
  t=8〜11(行2処理中):  lbA = [I, J, K, L]  ← 行2に上書きされていく
                                               (行1は lbB に押し出される)
```

**ラインバッファは「ベルトコンベアの待機棚」のようなもの。** ピクセルが左から右へ流れるコンベアに乗って来る。棚に1行分を溜めておくと、次の行のピクセルが同じ列に来たとき、棚から1行前の値を読み出して一緒に処理できる。

#### 3.3.4 列シフトレジスタ — 横方向3列分を揃える

ラインバッファで **縦方向** は解決した（3行分を同時取得）。次は **横方向** — 3列分を揃える必要がある。

ラインバッファの出力 (prev2, prev1, in_pixel) は「今アクセスした列」の3行分しか出さない。しかし 3x3 ウィンドウには「今の列」「1つ前の列」「2つ前の列」の3列が必要。

解決策: **各行にシフトレジスタを置いて、過去2クロック分の値を保持する。**

```
ラインバッファ出力                   シフトレジスタ（3段）
                              2つ前      1つ前      最新
                            +--------+  +--------+  +--------+
prev2 (2行前) ------------->| w[0][0]|<-| w[0][1]|<-| w[0][2]|<-- prev2
                            +--------+  +--------+  +--------+
prev1 (1行前) ------------->| w[1][0]|<-| w[1][1]|<-| w[1][2]|<-- prev1
                            +--------+  +--------+  +--------+
in_pixel (現在行) --------->| w[2][0]|<-| w[2][1]|<-| w[2][2]|<-- in_pixel
                            +--------+  +--------+  +--------+
```

毎クロック、新しい値が右端に入り、全部が1つずつ左にシフトする。

##### 具体的に追跡する（4列幅画像、行2を処理中）

```
画像:
  行0:  A  B  C  D
  行1:  E  F  G  H
  行2:  I  J  K  L
```

**t=8: I（行2, 列0）到着**
ラインバッファ: prev2=A, prev1=E, in_pixel=I

```
シフトレジスタ:  w[0] = [?, ?, A]    w[1] = [?, ?, E]    w[2] = [?, ?, I]
                        古い  古い 最新          古い  古い 最新          古い  古い 最新
```

ウィンドウは不完全（列0が初めてなので左側が不定）。まだ出力しない。

**t=9: J（行2, 列1）到着**
ラインバッファ: prev2=B, prev1=F, in_pixel=J

```
シフトレジスタ: 左シフトして新値を右端に入れる
  w[0] = [?, A, B]    w[1] = [?, E, F]    w[2] = [?, I, J]
```

まだ左端が不定。もう1クロック待つ。

**t=10: K（行2, 列2）到着** ★最初の有効ウィンドウ★
ラインバッファ: prev2=C, prev1=G, in_pixel=K

```
シフトレジスタ: 左シフトして新値を右端に入れる
  w[0] = [A, B, C]    w[1] = [E, F, G]    w[2] = [I, J, K]
```

これで完全な 3x3 ウィンドウが揃った！

```
w[0][0]=A  w[0][1]=B  w[0][2]=C     ← 行0、列0-2
w[1][0]=E  w[1][1]=F  w[1][2]=G     ← 行1、列0-2
w[2][0]=I  w[2][1]=J  w[2][2]=K     ← 行2、列0-2
```

中心ピクセル = `w[1][1]` = **F**。 **出力はピクセル F に対する畳み込み結果。**
（入力は K だが、出力は F — 1行+1列分の遅延）

**t=11: L（行2, 列3）到着**

```
シフトレジスタ:
  w[0] = [B, C, D]    w[1] = [F, G, H]    w[2] = [J, K, L]
```

中心 = `w[1][1]` = **G**。次の畳み込み出力。

#### 3.3.5 まとめ — 2つの仕組みの組み合わせ

```
                    ラインバッファ                シフトレジスタ
何を解決するか？    縦方向（行をまたぐ）          横方向（列をまたぐ）
保存する量          画像幅 x 行数分               3列分 x 3行
保存場所            BRAM（大容量・低コスト）      フリップフロップ（9個）
動作                同じ列が再来したとき          毎クロック左シフト
                    古い値を読み出し新値を書く
```

この2つを組み合わせることで、1クロックに1ピクセルしか届かないストリームから、毎クロック完全な 3x3 近傍ウィンドウを構成できる。

#### 3.3.6 なぜ BRAM を使うのか（フリップフロップとの比較）

ラインバッファの棚は画像幅分（640個）必要。各棚に24ビット（RGB888）を保存する。

| 実装方式 | 必要リソース（バッファ1つ） | Z-7020 の予算に対する割合 |
|---------|--------------------------|-------------------------|
| BRAM | 36Kb タイル 1個 | 140個中の1個 = **0.7%** |
| フリップフロップ | 640 x 24 = 15,360 FF | 106,400個中の15,360個 = **14.4%** |

BRAM なら微小なコストで済む。RTL で BRAM を使うための鍵:

```systemverilog
(* ram_style = "block" *) logic [23:0] lbA [0:LINE_PIXELS-1];  // 属性 -> BRAM
always_ff @(posedge clk) begin       // このブロックにリセットを書かない（重要！）
    if (in_valid) begin
        prev1    <= lbA[col];        // 読出
        lbA[col] <= in_pixel;        // 書込（同一アドレス、同一サイクル）
    end
end
```

**罠: 非同期リセットが BRAM 推論を破壊する。** `if (!rst_n) for (i) lbA[i] <= 0;` と書くと、Vivado は配列を BRAM にマッピングできない（BRAM は内容の非同期リセットをハードウェア的に持たない）。結果、約15,000個のフリップフロップとして合成され、リソースを浪費する。

#### 3.3.7 5x5 への拡張

5x5 カーネルでは同じ原理を拡張する:
- **4つの BRAM ラインバッファ** (lbA..lbD) で行 N-1 〜 N-4 を格納
- **5段の列シフトレジスタ** で `w[0:4][0:4]` を形成
- 中心ピクセル = `w[2][2]`

### 3.4 DSP48 積和演算 — 演算の心臓部

#### 3.4.1 毎クロックの計算量

ウィンドウ `w[0:2][0:2]` が形成されたら、3つの RGB チャネルそれぞれで独立に畳み込み和を計算する:

```
acc = coeff[0]*w[0][0] + coeff[1]*w[0][1] + coeff[2]*w[0][2]
    + coeff[3]*w[1][0] + coeff[4]*w[1][1] + coeff[5]*w[1][2]
    + coeff[6]*w[2][0] + coeff[7]*w[2][1] + coeff[8]*w[2][2]
```

1チャネルあたり 9 回の乗算と 8 回の加算、x3チャネル = **毎クロック27回の乗算**。

CPU ならこの27回の乗算を逐次実行する（最低27クロックサイクル）。FPGA は27個の物理的な乗算回路を実体化し、すべてが**並列に**動作する — 27個の積が1クロックサイクルで計算される。

#### 3.4.2 DSP48E1 スライス — 専用シリコン乗算器

Zynq-7020 は 220 個の DSP48E1 スライスを含む。各スライスは硬化シリコンブロック（コンフィギュラブルファブリックではなく、算術演算に最適化された固定トランジスタ回路）:

```
+---------------------------------------------+
|                DSP48E1                       |
|                                              |
|  A[29:0] --+                                 |
|             +-- プリ加算器 --> 25x18 乗算器 --> 48bit アキュムレータ --> P[47:0]
|  B[17:0] --+     (A+/-D)                                                |
|  D[24:0] --+                                                            |
|  C[47:0] ----------------------------------------> (バイパス加算器) ----+
|                                                                          |
|  各サブステージに内部パイプラインレジスタ                                  |
+---------------------------------------------+
```

畳み込みでは、各 DSP48 が `係数 x ピクセルチャネル値` の1つを計算する:
- **A入力** = 係数（符号付き8ビット、25ビットに符号拡張）
- **B入力** = ピクセルチャネル値（符号なし8ビット、18ビットにゼロ拡張）
- **P出力** = 符号付き17ビットの積

DSP48 はファブリッククロック（125 MHz）で動作し、LUT を一切消費せずに毎サイクル1つの積を生成する — 乗算はコンフィギュラブルファブリックではなく、専用トランジスタ論理で行われる。

#### 3.4.3 なぜ DSP48 であり LUT 乗算器ではないのか？

LUT で構築した 8x8 符号付き乗算器は約60 LUT とキャリーチェーンロジックを消費する。27個の乗算器で約1,620 LUT を使い、密な局所配線が発生する。Z-7020 での実測:

| 方式 | LUT コスト | 配線 | タイミング (WNS) |
|------|-----------|------|-----------------|
| LUT 乗算器 (27個) | ~1,620 LUT | 重度の混雑 | **-0.925 ns（不合格）** |
| DSP48 スライス (27個) | ~0 LUT | クリーン | **+0.125 ns（合格）** |

`(* use_dsp = "yes" *)` 合成属性で Vivado に乗算を DSP48 にマッピングさせる:

```systemverilog
(* use_dsp = "yes" *) logic signed [16:0] prod [0:2][0:8];
// prod[チャネル][タップ] = 係数[タップ] * ピクセルチャネル値
// 3チャネル x 9タップ = 27 DSP48 スライス
```

#### 3.4.4 符号付き x 符号なし乗算

係数は符号付き（例: Sobel は -1, -2 を含む）だが、ピクセル値は符号なし（0〜255）。RTL では明示的な符号拡張で対処する:

```systemverilog
prod[sc][t] <= coef(t) * $signed({1'b0, tap(t, sc)});
//              ^ 符号付き 8ビット    ^ 符号なし 8ビット、符号付き 9ビットにゼロ拡張
//              (-128..+127)          (0..255 -> 符号付き 0..255)
```

`{1'b0, pixel}` は0ビットを先頭に付加し、符号なし8ビット [0, 255] を符号付き9ビット [0, 255] に変換する。積は符号付き17ビット（範囲: -128x255 = -32,640 〜 +127x255 = +32,385）。

### 3.5 パイプライン・ステージング — 125 MHz でタイミングを閉じる

#### 3.5.1 なぜパイプラインが必要か？

125 MHz では1クロックサイクルが 8 ns。ウィンドウ形成→9回の乗算→9入力の合計→シフト→クランプの全計算が純粋な組合せ論理だとすると、伝搬遅延は 8 ns を大きく超える。解決策は計算を**パイプラインステージ**に分割し、各ステージを1クロック周期内に収めること:

```
     ステージ1     ステージ2     ステージ3     ステージ4     ステージ5
     (1サイクル)   (1サイクル)   (1サイクル)   (1サイクル)   (1サイクル)
     +--------+    +--------+    +--------+    +--------+    +---------+
入力>|ライン  | -> |ウィンドウ| -> |27 DSP  | -> |9入力   | -> |シフト   | -> 出力
     |バッファ|    |シフト   |    |乗算    |    |加算    |    |+飽和    |
     |読出    |    |レジスタ |    |(全並列) |    |(合計)  |    |         |
     +---+----+    +---+----+    +---+----+    +---+----+    +---+-----+
         |             |             |             |             |
         v             v             v             v             v
      レジスタ      レジスタ      レジスタ      レジスタ      レジスタ
      (FF)          (FF)          (FF)          (FF)          (FF)
```

各ステージの出力はフリップフロップレジスタに保持される。隣接するレジスタ間の組合せ論理は 8 ns 以内に完了できるほど短い。

**スループット**: 5ステージのレイテンシがあるにも関わらず、新しいピクセルが毎クロックステージ1に入り、完成したピクセルが毎クロックステージ5から出る。パイプラインは毎秒1.25億ピクセルを処理する — 必要な約920万ピクセル/秒（640x480x30fps）を大きく上回る。

**レイテンシ**: 各ピクセルの結果は入力から5クロックサイクル後に出現する。この固定遅延は視聴者には不可視（40 ns）であり、制御マーカー（sof, eol, eof, err）を同数のステージを持つマッチング・シフトレジスタで遅延させることで補償する。

#### 3.5.2 マーカー遅延チェーン

フレーム/ラインマーカーは、対応するピクセルと同一クロックサイクルで出力に到着しなければならない。並列シフトレジスタで同数ステージ分遅延させる:

```systemverilog
// マーカーはピクセルデータと並行して遅延させる
always_ff @(posedge clk) begin
    {v2, s2, e2, f2, r2} <= {v1, s1, e1, f1, r1};   // ステージ2
    {v3, s3, e3, f3, r3} <= {v2, s2, e2, f2, r2};   // ステージ3
    {v4, s4, e4, f4, r4} <= {v3, s3, e3, f3, r3};   // ステージ4
    // ... 全ステージを通して
end
// 最終出力: out_valid <= v4; out_sof <= s4; ...
```

これにより `out_sof`（フレーム開始）は常にそのフレームの最初の処理済みピクセルと正確に同一クロックサイクルに出現する。

#### 3.5.3 5x5 の加算木問題

5x5 畳み込みでは、25個の積を1つのクロックステージで合計すると 25入力加算になり、8 ns に収まらない。解決策は **2段加算木**:

```
ステージ4 (グループ和):   5個ずつ5グループの部分和
   group[0] = prod[0] + prod[1] + prod[2] + prod[3] + prod[4]
   group[1] = prod[5] + prod[6] + prod[7] + prod[8] + prod[9]
   ... (各5入力加算の部分和 x 5)
                          |
                       レジスタ (FF)
                          |
ステージ5 (累積):  acc = group[0] + group[1] + group[2] + group[3] + group[4]
                          |
                       レジスタ (FF)
```

各ステージは5値のみ加算（4回の加算）するので 8 ns に収まる。25値の合計を2クロックサイクルで計算する。

### 3.6 正規化と飽和

積の合計の後:

```
v = accumulated_sum >>> cfg_shift    // 算術右シフト（符号を保持）
if (cfg_abs && v < 0)  v = -v       // 絶対値（勾配の大きさ用）
out = (v < 0) ? 0 : (v > 255) ? 255 : v[7:0]   // [0, 255] に飽和
```

`cfg_shift` パラメータでカーネルを正規化する。ガウシアン `{1,2,1; 2,4,2; 1,2,1}` の係数合計は 16 なので、`shift = 4`（16で割る）で輝度を保持する。

**なぜ除算でなく算術右シフト (`>>>`) か？** 除算はハードウェアでコストが高い（反復型または大規模な組合せ除算器）。N ビットの右シフトは 2^N で割ることと等価 — そしてハードウェアでは右シフトは**タダ**: 上位ビットを出力に接続し、下位 N ビットを捨てるだけ。論理ゲートを一切消費しない。畳み込みカーネルの正規化係数が2の累乗（16, 256 等）で設計されるのはこのため。

**飽和**は結果を有効な符号なし8ビット範囲 [0, 255] にクランプする。クランプしないと、`(4 x 255) >>> 0 = 1020` が `252`（1020 mod 256）にラップアラウンドし、不正な明るいピクセルが生じる。クランプ論理は単純な比較器:

```systemverilog
sat = (v < 0) ? 8'd0 : (v > 255) ? 8'd255 : v[7:0];
```

### 3.7 エッジ検出のための絶対値 (`cfg_abs`)

Sobel 等のエッジ検出カーネルは**符号付き**の出力を生成する: 一方の勾配方向で正、他方で負。`cfg_abs` なしでは、負の値は 0 にクランプされ、エッジの半分が失われる。

`cfg_abs = 1` の場合、最終的な [0, 255] 飽和の**前に**絶対値を取る:
- Sobel-X (abs なし): 明→暗方向の水平エッジのみ検出
- Sobel-X (abs あり): 両方向の水平エッジを検出

これは全方向エッジ強度モード（第6章参照）に不可欠。

## 4. 一般 5x5 畳み込み (`axis_rgb_conv5x5`)

### 4.1 原理

5x5 カーネルは 25 個の係数を持ち、より広い空間サポートを実現する — 広い半径のガウシアンぼかし、または Difference of Gaussians の片方として有用。

### 4.2 2段加算木

25個の積を1クロックサイクルで合計すると非常に長い組合せパスが生じる。5x5 モジュールは **2段加算木**を使用する:

```
ステージ4 (グループ和):  5個ずつ5グループの積 -> 5個の部分和
ステージ5 (累積):        5個の部分和 -> 1つの合計
```

25入力の合計を5入力の2ステージに分割し、それぞれが 125 MHz のタイミングを満たす。

### 4.3 リソースコスト

- 75 DSP48 スライス（25タップ x 3 RGBチャネル）
- 4 BRAM ラインバッファ（各 640 x 24ビット）
- 6パイプラインステージ

## 5. Difference of Gaussians (DoG) — 二重カーネル・コンバイナ

### 5.1 原理

**Difference of Gaussians (DoG)** は、広い半径のガウシアン（ぼかし大）から狭い半径のガウシアン（ぼかし小）を引くことで形成されるバンドパスフィルタ:

```
DoG(x, y) = G_small(x, y) - G_large(x, y)
```

これは Laplacian of Gaussian (LoG) の近似であり、コンピュータビジョンの標準的手法:
- **ブロブ検出**: 周囲と異なる領域を検出
- **エッジ検出**: DoG のゼロ交差がエッジに対応
- **アンシャープマスキング**: ぼかし版を引くことでディテールを強調

### 5.2 アーキテクチャ — 並列分岐

3x3 と 5x5 の畳み込みモジュールが**同一の入力ピクセルストリーム**を並列に処理する:

```
入力 ----+---- conv3x3 (小カーネル, A) ---- 高速パス (5ステージパイプライン)
         |
         +---- conv5x5 (大カーネル, B) ---- 低速パス (6ステージパイプライン)
```

両方とも入力ピクセルごとに1つの出力ピクセルを同一ラスタ順で生成する。A の k 番目の出力と B の k 番目の出力は**同一空間ピクセル**に対応する。B は単に A よりも固定レイテンシ分（約1ライン+1サイクル、追加ラインバッファのため）遅れる。

### 5.3 序数整列 FIFO

コンバイナはレイテンシ差を算術的に計算しようとしない（脆弱でエラーを招く）。代わりに**序数整列 FIFO** を使用する:

- A ピクセルは `a_valid` で BRAM FIFO に**プッシュ**される
- A ピクセルは `b_valid` で FIFO から**ポップ**される

両方の分岐が同一順序でピクセルを処理するため、k 番目のポップは常に k 番目のプッシュに一致する — レイテンシ計算なしの自動整列。

FIFO 深さ（1024エントリ）は A の B に対する最大リード（640ピクセル/ライン + パイプライン差 ≈ 641エントリ）を超えればよい。

### 5.4 結合式

```
out = sat( (alpha * A - beta * B) >>> shift + offset )
```

4つのモード:
- **mode 0**: A パススルー（3x3のみ）
- **mode 1**: B パススルー（5x5のみ）
- **mode 2**: DoG = alpha*A - beta*B（バンドパス / エッジ / ブロブ）
- **mode 3**: Sum = alpha*A + beta*B（マルチスケールブレンド）

`alpha`、`beta`（符号なし8ビット）、`shift`（4ビット）、`offset`（符号付き9ビット）はすべてランタイム設定可能。

### 5.5 実用プリセット

| プリセット | 3x3 カーネル | 5x5 カーネル | alpha | beta | shift | offset | 効果 |
|-----------|-------------|-------------|-------|------|-------|--------|------|
| blob | ガウシアン 3x3 | ガウシアン 5x5 | 1 | 1 | 0 | 128 | バンドパス・ブロブ検出 |
| unsharp | 恒等 | ガウシアン 5x5 | 2 | 1 | 0 | 0 | ディテール強調 |

## 6. 全方向エッジ強度 (`|Gx| + |Gy|`)

### 6.1 単一カーネル・エッジ検出の限界

単一の 3x3 Sobel-X カーネルは**水平エッジ**（垂直方向勾配）のみを検出する。同様に Sobel-Y は垂直エッジのみ。単一カーネルでは全方向のエッジを捉えられない。

真のエッジ強度は:

```
M = sqrt(Gx^2 + Gy^2)    （ユークリッドノルム、ハードウェアでは高コスト）
```

平方根を避ける一般的な近似:

```
M ≈ |Gx| + |Gy|          （L1ノルム、低コスト、最大 sqrt(2) 倍の過大評価）
```

### 6.2 並列畳み込み + 絶対値 + 和による実装

既存の二重カーネル・アーキテクチャを再利用する:

```
分岐A (conv3x3):  Sobel-X カーネル, cfg_abs = 1  -->  |Gx|
分岐B (conv5x5):  Sobel-Y カーネル (5x5に中央埋込), cfg_abs = 1  -->  |Gy|
コンバイナ (mode 3 = 和):  alpha=1, beta=1  -->  |Gx| + |Gy|
```

**鍵となる知見**: 絶対値は各畳み込みモジュール内部（[0, 255] 飽和の前）で計算しなければならず、コンバイナで行ってはならない。飽和後に abs を適用すると、負の勾配値はすでに 0 にクランプされており、エッジ情報の半分が不可逆に失われる。

`cfg_abs` が畳み込みモジュールごとの制御である理由:
- conv3x3 の `cfg_abs`: `v = acc >>> shift; if (v < 0) v = -v; clamp [0, 255]`
- conv5x5 の `cfg_abs`: 同上
- コンバイナは非負の [0, 255] 値のみを受け取るので、`sum` モードが `|Gx| + |Gy|` を生成する

### 6.3 検証

カラーバーテストパターン: 単一 Sobel-X（abs なし）はエッジ% = 1.1%。全方向（両方 abs あり、sum モード）ではエッジ% = 2.2% — 正確に2倍で、両極性の勾配が回復されたことを確認。

## 7. カスケード可変サイズ・ガウシアンぼかし

### 7.1 原理 — 分離型畳み込み

2次元ガウシアンカーネルは**分離可能**: 2つの1次元ベクトルの外積に分解できる:

```
G_2D = g_v * g_h^T
```

ここで `g_h` と `g_v` は1次元ガウシアンベクトル。5x5 ガウシアンを2回の1次元パスで計算できる:

```
パス1 (水平):  row_filtered(x, y) = Σ(j) h[j] * image(x+j, y)
パス2 (垂直):  out(x, y) = Σ(i) v[i] * row_filtered(x, y+i)
```

**コスト: 5+5 = 10 乗算/チャネル（25の代わり）。** 3チャネルで: 30 DSP vs 75 DSP。Z-7020 の 220 DSP 予算内で複数ステージのカスケードが可能になる。

### 7.2 カスケード・アーキテクチャ

3つのぼかしステージを直列接続:

```
conv5x5 (S1, 一般型) --> conv5x5_sep (S2, 分離型) --> conv5x5_sep (S3, 分離型)
```

各ステージは独立に恒等（パススルー）またはガウシアンカーネルに設定可能。組み合わせにより:

| 有効ステージ | 実効カーネルサイズ | 使用 DSP |
|-------------|-------------------|----------|
| S1 のみ | 5x5 | 75 |
| S1 + S2 | 9x9 | 75 + 30 = 105 |
| S1 + S2 + S3 | 13x13 | 75 + 60 = 135 |

### 7.3 カスケードで実効サイズが増大する理由

2つのガウシアンフィルタを逐次適用すると、分散が**加算**される:

```
sigma_combined^2 = sigma_1^2 + sigma_2^2
```

空間的には実効カーネル幅も増大する。5x5 ガウシアンに別の 5x5 ガウシアンを適用すると、単一の約 9x9 ガウシアンと等価になる（幅5の2つの関数の畳み込みはサポート 5+5-1 = 9 を持つ）。3ステージで 5+5+5-2 = 13。

### 7.4 分離型モジュールの内部構成

`axis_rgb_conv5x5_sep` モジュールは8ステージパイプライン:

**水平パス (ステージ1-4):**
1. 5段の列シフトレジスタ (1クロック)
2. 15 DSP 積: `h_coeff[i] * pixel_channel` (1クロック)
3. 5入力の水平合計 (1クロック) — シフト/クランプから分離してタイミング修正
4. 符号付き12ビットに再量子化（設定可能なシフト）(1クロック)

**垂直パス (ステージ5-8):**
5. パックされた36ビット水平結果の4つの BRAM ラインバッファ (1クロック)
6. 15 DSP 積: `v_coeff[j] * h_result_row_j` (1クロック)
7. 5入力の垂直合計 (1クロック) — シフト/飽和から分離
8. 正規化 + 符号なし8ビットに飽和 (1クロック)

中間の水平結果は符号付き12ビット（8ビットではなく）で保持し、2パス計算を通じて精度を保存する。

## 8. メディアンフィルタ (3x3)

### 8.1 原理

**メディアンフィルタ**は各ピクセルをその 3x3 近傍の中央値（ソートしたときの中間値）で置換する。線形フィルタ（ガウシアン）と異なり:

- エッジを保存する（シャープな境界を越えたぼかしなし）
- **ごま塩ノイズ**（孤立した極端値）を効果的に除去
- 非線形である（畳み込みカーネルとして表現できない）

### 8.2 ソーティングネットワーク (median9)

9個の値の中央値を見つけるには、**ソーティングネットワーク** — 比較交換 (CAS) 操作の固定シーケンスが必要:

```
CAS(a, b) = { min(a,b), max(a,b) }
```

実装は **Smith/Paeth median-of-9 ネットワーク**を使用: 19 CAS 操作が9つの依存層に配置される。125 MHz タイミングを満たすため、5パイプラインステージ（ステージあたり最大2直列 CAS）に分割:

```
ステージ1: CAS(1,2) CAS(4,5) CAS(7,8) then CAS(0,1') CAS(3,4') CAS(6,7')
ステージ2: CAS(1,2) CAS(4,5) CAS(7,8) then CAS(0,3) CAS(5,8) CAS(4,7)
ステージ3a: CAS(3,6) CAS(1,4) CAS(2,5)
ステージ3b: CAS(4,7) then CAS(2,4')
ステージ4: CAS(4,6) then CAS(2,4') --> インデックス4に中央値
```

このネットワークは9値すべてを完全ソートしない — 第5順序統計量（中央値）のみを決定する。これで十分であり、より低コスト。

**正当性検証**: Knuth の 0-1 原理（全512バイナリ入力ベクトル）、9個の異なる値の全 362,880 順列、300,000 ランダムベクトルで検証済み。

### 8.3 チャネル別適用

3つの独立した `median9` インスタンスが R, G, B チャネルに並列で動作する。中央値は輝度ではなくチャネルごとに計算 — デノイズしつつ色情報を保存する。

### 8.4 レイテンシ整合

プリフィルタモジュールには3つの分岐（メディアン、ガウシアン、点処理）があり、すべてが同時に出力を生成しなければならない。全分岐がウィンドウ形成後5サイクルの等しいレイテンシにパディングされ、マーカーはマッチングするシフトレジスタチェーンで遅延させる。

## 9. ガウシアンぼかし (3x3, DSPゼロ)

### 9.1 原理

PRE ステージには **DSP スライスを使わない**固定 3x3 ガウシアンが含まれる — シフトと加算のみ:

```
カーネル:  [1, 2, 1]     重み:  コーナー x1、エッジ x2、中心 x4
           [2, 4, 2]     合計 = 16、>> 4 で正規化
           [1, 2, 1]
```

### 9.2 シフト加算による実装

重み (1, 2, 4) はすべて2の累乗なので、乗算をビットシフトで置換できる:

```
チャネルごと:
  corners = w[0][0] + w[0][2] + w[2][0] + w[2][2]         (4回の加算, x1)
  edges   = w[0][1] + w[1][0] + w[1][2] + w[2][1]         (4回の加算, x2)
  centre  = w[1][1]                                         (x4)
  total   = corners + (edges << 1) + (centre << 2)          (シフト + 加算)
  output  = total >> 4                                      (切り捨て = /16)
```

乗算器を一切使わない。4ステージパイプラインでメディアン分岐のレイテンシと一致させる。

## 10. ディザリング — ビット深度削減（POST の後段）

### 10.1 原理

最終ステージ（`axis_rgb_dither`、POST 点処理スロットの後・キャプチャブリッジの前）は、各チャネルを
N ビットに削減する際、量子化の**前に**位置依存またはノイズ依存のバイアスを加える。バイアスが画素を
量子化境界をまたぐよう構造的に押すため、なだらかな勾配がハードなバンドではなくディザパターンとして
描画される。用途: N=1 ハーフトーン(0/255)、N=2〜4 ポスタライズ/レトロ、N=6 低ビットパネルのバンド抑制。
ステートレス（ラインバッファ無し）— 数十 LUT、固定1サイクルレイテンシ。`cfg_ctrl=0`（0xFE4A）=
パススルー（ビット一致の既定）。

### 10.2 順序ディザ（Bayer 4x4）

`(x%4, y%4)` で参照する 4x4 Bayer 閾値行列が、決定論的でタイル可能なバイアスを与える:

```
   0  8  2 10
  12  4 14  6     値 0..15 (/16)
   3 11  1  9
  15  7 13  5
```

行列値を切り捨てる下位ビット範囲にスケールして加算 → 固定の規則的な網点パターン（古典的な「順序
ディザ」の見た目）。ラインバッファもフィードバックも不要。

### 10.3 ランダムディザ（LFSR）

8ビット Galois LFSR（有効画素ごとに1回進める）が、下位ビット範囲にマスクした擬似乱数バイアスを供給
→ 白色ノイズディザ（粒状・パターンレス）。LFSR 1個、コストは無視できる。

### 10.4 量子化 + フルレンジ複製

```
drop = 8 - N
bias = 順序: bayer << (drop-4)        |   ランダム: lfsr & (2^drop - 1)
v    = clamp(in + bias, 255)
q    = v & ~(2^drop - 1)              (上位 N ビットを残す)
out  = q | q>>N | q>>2N | q>>4N       (MSB を複製してフル 8 ビット範囲へ)
```

複製（ビットスメア）で N ビットのコードをフル 0..255 範囲に写すため、N=1 → {0,255}（0/128 ではない）、
N=2 → {0,85,170,255} となる。位置 `(x,y)` は内部の col/row カウンタから（`col` は conv3x3 と同じ、
`row` カウンタを追加: EOL でインクリメント、EOF でリセット）。

### 10.5 なぜ順序ディザで、誤差拡散ではないのか

Floyd–Steinberg / Atkinson の誤差拡散は最高画質（パターンレス）だが、全幅ラインバッファと画素ごとの
左画素誤差フィードバックループが必要 — 追加ロジックと、すでに混雑している sysclk 上のタイトな
フィードバック経路になる。順序＋ランダムはステートレスで難なくタイミングが閉じる。誤差拡散は保留。
実機検証済（WNS +0.112）: ハーフトーンの Bayer 網点が HDMI でライブ表示され、5 モード
（OFF / ハーフトーン / poster-2 / poster-4 / random）がモニタ上で巡回する。

## 11. ランタイム設定

全処理パラメータは **0xFE SCCB インターセプト**機構でランタイム設定可能: SCCB エンジンがアドレス `0xFExx` への書き込みを受信すると、データをカメラチップに送信する代わりに内部係数レジスタにルーティングする。既存の SCCB 書き込みパス（AXI GPIO）を再利用するため、追加のブロックデザイン変更は不要。

| アドレス範囲 | 対象 |
|-------------|------|
| `0xFE00-08` | 3x3 カーネル係数（9 x 符号付き8ビット） |
| `0xFE09` | 3x3 正規化シフト |
| `0xFE20-38` | 5x5 カーネル係数（25 x 符号付き8ビット） |
| `0xFE39` | 5x5 正規化シフト |
| `0xFE40-44` | DoG コンバイナ（モード, alpha, beta, shift, offset） |
| `0xFE45` | 絶対値有効化（bit0=3x3, bit1=5x5） |
| `0xFE46-49` | PRE/POST 処理モードと閾値 |
| `0xFE4A` | ディザ制御（[0]=有効, [1]=モード 0 順序/1 ランダム, [4:2]=bits/ch） |
| `0xFE50-5B` | カスケード S2 分離型カーネル（h[5], hshift, v[5], vshift） |
| `0xFE60-6B` | カスケード S3 分離型カーネル |

`cfg_proc_op[3:0]` フィールド（IDELAY GPIO ワードのビット [24:21]）が最終マルチプレクサの出力を選択する。

## 13. 設計上の教訓

1. **乗算は DSP に載せよ**: Z-7020（LUT 配線が逼迫）では畳み込みの乗算は必ず DSP48 スライス（`(* use_dsp = "yes" *)`）を使う。LUT ベースの乗算は 125 MHz でタイミング不合格になる。

2. **長い組合せパスを分割せよ**: 合計 + シフト + クランプを1ステージに詰めると WNS 違反が発生する。レジスタで分離（合計 | シフト+クランプ）すれば確実にタイミングが閉じる。

3. **絶対値は飽和の前に取れ**: 勾配ベースの処理では、|v| は [0, 255] クランプの前に計算しなければならない。クランプ後は負の値がすでに 0 になっており回復不能。

4. **BRAM 推論にはリセットなし RAM 配列**: 配列宣言に非同期リセット（`always_ff ... if (!rst_n) for(...) mem[i] <= 0`）を書くと、Vivado は配列をフリップフロップ（約24k FF）として合成し BRAM にならない。解決策: RAM 読書きはリセットなしの `always_ff` ブロックに、ポインタ/制御のみリセットありブロックに。

5. **分岐整列には序数 FIFO**: 並列分岐間のレイテンシ差を正確に計算するのは脆弱。序数 FIFO（A の valid でプッシュ、B の valid でポップ）ならレイテンシ計算不要で自動整列。

6. **分離型カーネルで DSP を節約**: 分離型 5x5 は 30 DSP、一般型 5x5 は 75 DSP — 小型 FPGA の DSP 予算内でカスケード・アーキテクチャが可能になる。
